## Clarification: `.agents-output/` vs `docs/prompts/tasks/`

These are **separate concerns**:

- `docs/prompts/tasks/` — AI-generated planning docs for tasks provided by a human (`task-001` to `task-013`). Written before implementation, checked into version control as design artifacts.
- `.agents-output/` — runtime pipeline artifacts generated during agent runs (user requests, generated specs, test results). Created on-the-fly; not human-authored, but human-validated files.

The agents should write into `docs/prompts/tasks/` a folder per task with the following structure:

```plaintext
.docs/prompts/tasks/
                |__ task-010-x-content-generation/
                  |__ README.md <== store the specifications for the task 10 generated by AI
                |__ task-011-linkedin-content-generation/
                  |__ README.md <== store the specifications for the task 11 generated by AI
                ...
                |__ task-NNN-[slug of taks title read from github]/
                  |__ README.md <== store the user-request
                  |__ business-specifications.md
                  |__ technical-specifications.md
                  |__ test-results.md
```

For tasks 10 to 13, DON'T run the specifications agent. However, the other agents should update the `README.md` in the task subfolder as the job is executed.

Beyond those tasks, the agents should use the `task-NNN` structure as template. From then, the logic of tasks 10 to 13 MUST be removed from agents brain.

## agent-2-coder.md

**Replace Python examples** in the Object Calisthenics section with TypeScript equivalents:

### no-else rule (TypeScript)

Before:

```typescript
function statusLabel(code: number): string {
  if (code === 200) {
    return 'ok'
  } else {
    return 'not ok'
  }
}
```

After (guard clause, no `else`):

```typescript
function statusLabel(code: number): string {
  if (code === 200) return 'ok'
  return 'not ok'
}
```

### one-level-of-indentation rule (TypeScript)

Before:

```typescript
function collectValid(items: Item[]): Item[] {
  const result: Item[] = []
  for (const item of items) {
    if (item.isValid()) {
      result.push(item)
    }
  }
  return result
}
```

After (inner block extracted):

```typescript
function collectValid(items: Item[]): Item[] {
  return items.filter(isValid)
}

function isValid(item: Item): boolean {
  return item.isValid()
}
```

**Replace** "Python standard library's own conventions (e.g. subclassing `html.parser.HTMLParser`)" with the TypeScript/Vue equivalent:

> Where strict compliance would conflict with framework conventions (e.g. Vue lifecycle hooks, composable conventions following `useXxx` patterns), document the exception in the technical-choices section.

**Add:** If the implementation introduces an architectural decision not yet documented in `docs/decisions/`, add `### ADR Required` to the technical-spec file. The orchestrator must prompt the human to approve the ADR before committing code.
